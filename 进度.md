## 阶段 7a：BoxesZero 反向训练 ✅ 已完成

参考论文: BoxesZero: An Efficient and Computationally Frugal Dots-and-Boxes Agent (Entropy 2025)

### 核心算法

**反向训练 (Backward Training)**: 从终局开始学习，逐步向开局推进
- **阶段1 (Value Network Reinforcement)**: st=50→0，每步 MCTS 自对弈只搜 st 之后的步骤
  - st 之前: 贪心启发式 (优先吃格>安全边>危险边)
  - st 之后: MCTS 搜索 + 收集数据
  - 价值目标: root Q (MCTS Q值)
  - 晋级条件: mqr >= threshold 或 迭代次数 >= mit
- **阶段2 (Policy Network Reinforcement)**: st=0，全程 MCTS
  - 价值目标: 0.25×Q + 0.75×z (混合 Q 值和最终胜负)

### 实现文件

| 文件 | 说明 |
|---|---|
| `src/backward_selfplay_main.cpp` | 反向自对弈 CLI |
| `src/AI/az/az_selfplay.h/cpp` | 新增 `runBackward()`, `playOneGameBackward()`, `rootQ` |
| `train/backward_training.py` | 编排脚本: 自对弈→训练→导出→晋级 |
| `train/dataset.py` | 新增 `value_mode` (margin/wdl/q/q+z) |
| `train/train.py` | 新增 `--value_mode`, `--q_weight` 参数 |

### 用法

```bash

  --arch resnet_s \
  --games_per_iter 50 \
  --simulations 400 \
  --epochs_per_iter 30 \
  --start_st 50 --beta 5 --mit 3
```

## 阶段 7b：Bug 修复 + Arena 评估 + GUI 部署 ✅ 已完成（2025-05-18）

### 发现并修复的 Bug

1. **AlphaZeroMove 回合逻辑错误** (`az_move.cpp`)
   - 原问题: MCTS 落子后即使吃到格子，外层也强制换手
   - 修复: `AlphaZeroMove` 内部循环，吃到格子时同一玩家继续决策
   - 影响范围: GUI AI + evaluate_ai

2. **evaluate_ai 完全确定性** (`evaluate_ai_main.cpp`)
   - 原问题: 无 Dirichlet 噪声 + temperature=0 → 每局结果完全相同
   - 修复: 完全重写 `playOneGame`，支持 `--sims`/`--temp`/`--verbose`
   - 前几步加入轻微噪声 (alpha=0.3, frac=0.15) + 温度采样 (T=0.5)

3. **backward_training.py 数据覆盖** 
   - 原问题: 同一 st 的多轮 jsonl 文件名相同，复制到 combined 时被覆盖
   - 修复: 使用 `iter_{N:04d}__{filename}` 唯一命名
   - 注: 原 iter15 模型实际只用了 11779 样本（意外形成 sliding-window，效果反而不差）

4. **SFML ABI 不兼容** 
   - 原问题: 项目自带 SFML DLL (2022/GCC 12) 与当前 GCC 15.2 的 libstdc++ 不兼容
   - 修复: 从 GitHub 拉取 SFML 2.6.2 源码，用 GCC 15.2 重新编译

## 阶段 11：双模型雪崩诊断 + 回滚 iter22 + 协议修复（2026-05-26）

### TL;DR

iter23/iter24 黑方 0% 不是模型问题，而是 **三个独立 bug 叠加**：

1. **致命：MCTS 评估器路由 bug** — 双模型下 `getEvaluator()` 全局指针只在外层 turn 切换时设置一次。MCTS 1600 sims 中所有 white-to-move 内部节点都被错误地用 black 模型评估，反之亦然。黑/白模型从未见过对方颜色的训练样本，输出 garbage policy/value 污染整个搜索树。
2. **测量协议：fixed-color + 移动靶 baseline** — 记忆 c5900f29 证明在 N=20/side、temp=4 噪声下，**iter15 自己 vs 自己的黑方胜率仅 7.5%**。fixed-color 黑方门槛信号远小于结构性噪声 + 移动靶，注定不可超越。
3. **数据稀释** — 按 player 拆数据使每个网络的有效训练样本砍半，CNN 在 ~30K 样本下本就过拟合（阶段 4 已证），雪上加霜。

iter22 单 resnet_m + swap-color arena 是真正的局部最优。**回到 iter22，修 MCTS bug，换 score-margin 门槛**。

### 根因 1：MCTS 评估器路由 bug

**文件**：`src/AI/az/az_mcts.cpp` 第 62 / 146 行

```cpp
NetworkOutput rootOutput = getEvaluator().evaluate(board, player);     // 行 62
NetworkOutput output = getEvaluator().evaluate(node->board, node->player);  // 行 146
```

`getEvaluator()` 返回全局静态指针 `globalEvaluator`，它由外层 turn 切换调用 `selectEvaluatorForColor(currentPlayer)` 设置。MCTS 内部下降到不同颜色的叶节点时，**仍然用根节点颜色的评估器**。

**单模型下完全无害**（一个网络处理两种颜色 via 通道 5），但 dual 致命。

**修复**：在两处 `getEvaluator().evaluate()` 之前各加一行：

```cpp
selectEvaluatorForColor(node->player);  // 单模型模式下为 no-op
NetworkOutput output = getEvaluator().evaluate(node->board, node->player);
```

`selectEvaluatorForColor` 在 `blackEvaluator/whiteEvaluator` 全局指针为 null 时不修改 `globalEvaluator`（单模型 arena/GUI 路径完全不变）。

#### 真正瓶颈：Value Head 被 z（对局结果标签）拉偏

**现象链**：

```
selfplay 自对弈 → 白方赢 49/50 (98%)
  ↓
训练数据：黑方样本中 z = -1 占 98%
  ↓
value head 训练目标：黑方位置 ≈ -1.0（必输）
  ↓
学完后 root_q 显示：黑方 -0.74，白方 +0.75（双方认定白胜 75%）
  ↓
PUCT: score = Q + c·P·sqrt(N)/(1+N)
  Q 偏向极端值，更高 sims 让搜索更强地"相信"这个错误评估
  ↓
sims↑ → 黑方更"放弃"（Q 已 -0.74）→ 选最不痛的输法
sims↑ → 白方更"贪婪"（Q 已 +0.75）→ 选最大优势走法
  ↓
sims 翻 4 倍，黑方少赢 30pp（90%→60%）
```

**关键洞察**：policy network 知道怎么下（sims=800 时黑方 90%），但 value head 在拖后腿。搜索越深，错误的 value 越被放大。之前看 root_q ±0.75 觉得"MCTS 收敛了，sims 够了"，但**收敛到错的地方等于没收敛**。

#### 五个候选的最终判定

| 候选 | 之前判定 | 修订后判定 | 证据 |
|---|---|---|---|
| A: buffer 太小 | 重要 | 次要 | 1.65K 样本/iter × 8 windows = 13K 不算太少 |
| B: sims 不够 | 不是瓶颈 | **完全相反：sims 太多反而更糟** | 实验 sims=3200 |
| C: 单模型不对称 | 主要瓶颈 | 症状不是病因 | 不对称是 value bias 导致的 |
| D: 网络太小 | 不是瓶颈 | 不是瓶颈 | resnet_m 已能学好白方 |
| **E: value head 被 z 拉偏** ★ | — | **真正的瓶颈** | 三个实验一致指向 |

这与 BoxesZero (Entropy 2025) 论文结论一致：论文使用 root Q from MCTS 作为 value target，正是为了避免 "final game outcome z heavily biases the network toward the structurally favored player"。

#### 三重修复方案（已全部实现）

| 优先级 | 改动 | 工作量 | 预期效果 | 状态 |
|---|---|---|---|---|
| **P0** | `value_mode` q+z → q (纯 Q 目标) | 1 行默认值 | 立刻消除 value bias | ✅ 已实现 |
| **P1** | League selfplay（混合对手） | ~100 行 C++/Python | 注入黑方胜局数据 | ✅ 已实现 |
| **P2** | Dual model（默认开启） | 1 行默认值 | 隔离 color 干扰 | ✅ 已实现 |

**P0**：`--stage2_value_mode` 默认从 `"q+z"` 改为 `"q"`，`--q_weight` 默认 1.0。z = ±1 是 game-level 标签，对该局所有样本都一样→强烈 color bias；Q 是 MCTS 在具体局面的评估→position-level 无累积偏差。

**P1**：`--league` 开关，阶段 2 每轮拆分对局：70% 标准自对弈 + 20% 模型(黑) vs 启发式(白) + 10% 模型(黑) vs 旧模型(白)。C++ 支持单侧模型 (`--black-model` only → 另一侧自动启用启发式)。旧模型池自动维护（最多 5 个），状态持久化。

**P2**：`--dual` 默认 `True`（新增 `--no-dual` 回退）。双模型各自独立训练，黑方网络 value head 不被白方 98% 胜局污染。

## 阶段 15：MCTS 子树复用 + 渐进温度调度 + 更长自对弈（2026-06-13）

### 背景

iter23-32 的大规模实验证明模型确实在变强（margin 从 -3.52 到 +1.94），但被 5×5 白方结构性优势 + value head bias 锁住。当前瓶颈不在算法架构，而在：
1. **搜索效率低**：每步重建整棵树，大量重复计算
2. **数据质量可优化**：硬切换温度（T=1.0→0.0）产生不自然的探索-利用断层
3. **selfplay 步数偏少**：每局约 40 个决策点，数据深度不足

### 改动 1：MCTS 子树复用

**文件**：`src/AI/az/az_mcts.h`, `src/AI/az/az_mcts.cpp`

新增 `search(board, player, config, previousRoot, prevAction)` 重载：
- 将上一步所选动作对应的子树提升为新根节点
- 保留历史搜索信息（visits/valueSum），减少重复计算
- 自动检测棋盘是否因专家归一化而改变，不匹配时安全回退到新建树

**预期效果**：搜索速度提升 2-3 倍，相同 sims 下搜索质量更高

**注意**：由于 C 型格归一化可能改变棋盘，子树复用仅在标准落子步生效（无 C 型格被吃时棋盘不变）。专家归一化步骤会强制新建树，安全无副作用。

### 改动 2：渐进温度调度

**文件**：`src/AI/az/az_selfplay.cpp`

原逻辑：前 N 步 T=1.0，之后硬切 T=0.0
新逻辑：前 `temperatureMoves` 步内线性降温 `T = T_start * (1 - step/N)`

效果：更平滑的探索-利用过渡，训练数据分布更自然。

### 改动 3：温度参数暴露到 CLI

**文件**：`src/backward_selfplay_main.cpp`, `src/selfplay_main.cpp`

新增命令行参数：
- `--temperature F` — 初始温度（默认 1.0）
- `--temperature-moves N` — 降温步数（默认 10，0=全程贪心）

**Python 侧**：`backward_training.py` 新增 `--temperature` / `--temperature_moves` 参数，自动注入所有 selfplay 调用。

### 改动 4：ResNet-L 支持（已就绪）

`train/model.py` 阶段 4 已实现 `resnet_l`（10 blocks × 256 ch），本次无需额外修改。后续实验可直接使用 `--arch resnet_l`。

3. **ONNX Runtime 部署**：`onnxruntime.dll` + `onnxruntime_providers_shared.dll` 必须**同时**复制，少一个会报 LoadLibrary 126 错误
4. **网络策略**：裸连 GitHub 完全不可达（TCP 443 失败）；挂梯子后 `codeload.github.com` 立即可用
